# Introduction

Flows are the visual blueprints for your AI phone agents — defining what the agent says, collects, calls, and decides from “hello” to hang-up.

<Info>
  This page evolves as the product evolves. Start here, then jump to sections from the left nav as you need depth.
</Info>

## What are Flows?

<Cards>
  <Card title="Blocks" icon="sparkles">
    Individual units of behavior — speak to callers, collect info, call APIs, transfer, end calls, and more.
  </Card>

  <Card title="Edges" icon="arrow-right">
    Connections that determine which path the conversation takes next.
  </Card>

  <Card title="Variables" icon="tag">
    The agent’s short-term memory — values that change per call (caller inputs, system data, API results).
  </Card>
</Cards>

## Why use Flows?

<Cards>
  <Card title="No code to start">
    Drag, drop, and configure. Ship fast without waiting on engineering.
  </Card>

  <Card title="Real-time & natural">
    Handle interruptions, clarifications, and pivots mid-conversation.
  </Card>

  <Card title="Built to scale">
    Confidently handle thousands of concurrent calls.
  </Card>

  <Card title="Integrations">
    Call your APIs and systems directly from the Flow.
  </Card>
</Cards>

# Core concepts

## How to change a Flow

<Steps>
  <Step title="Enter edit mode">
    Open your Flow and click <b>
    Edit</b>

    .
  </Step>
  
  <Step title="Add blocks">
    Drag desired blocks from the left sidebar onto the canvas.
  </Step>
  
  <Step title="Configure blocks">
    Click a block to set prompts, fields, API details, and validation.
  </Step>
  
  <Step title="Connect blocks">
    Drag from a block’s output port (bottom) to the next block’s input (top).
  </Step>
  
  <Step title="Save">
    Click <b>
    Save</b>

    . Each save creates a new version.
  </Step>
</Steps>

<info>
  <b>
  Versioning:</b>

   Every save is a version. You can revert. Always test before pushing to production.
</info>

## Connecting blocks with edges

<Steps>
  <Step title="Create an edge">
    Click and drag from the output

     of one block to the input

     of the next.
  </Step>
  
  <Step title="Configure the edge">
    Select the edge to open the editor and set guards/requirements.
  </Step>
  
  <Step title="Confirm & save">
    Define what must be present or confirmed before the edge is taken, then save.
  </Step>
</Steps>

<Properties>
  <Property name="required_variables" type="array">
    Variables that must exist (and be non-empty) for the edge to be eligible.
  </Property>

  <Property name="confirm_variables" type="array">
    Variables the caller should confirm before taking the edge (e.g., <code>@name</code>, <code>@address</code>).
  </Property>
</Properties>

# Call variables

<Accordion>
  <AccordionItem title="What are call variables?">
  Variables are dynamic containers that persist for the duration of the call — like the agent’s working memory.
</AccordionItem>

  <AccordionItem title="How to set variables">
  Use <b>Collect</b> blocks for caller input, <b>API Request</b> blocks for external data, or add custom variables via the Variable modal.
</AccordionItem>

  <AccordionItem title="How to access variables">
  Reference by name with <code>@variable\_name</code> or select from the Variable modal. Use them in prompts, conditions, and API payloads.
</AccordionItem>
</Accordion>

### Variable types

<Properties>
  <Property name="text" type="string">General text.</Property>
  <Property name="number" type="number">Numeric values.</Property>
  <Property name="date" type="ISO date">Dates.</Property>
  <Property name="time" type="HH:mm">Times.</Property>
  <Property name="name" type="string">Caller names.</Property>
  <Property name="email" type="string">Email addresses.</Property>
  <Property name="phone" type="string">Phone numbers.</Property>
  <Property name="url" type="string">Web addresses.</Property>
  <Property name="address" type="object">Street, city, region, postal, country.</Property>
  <Property name="currency" type="number">Monetary values.</Property>
  <Property name="percentage" type="number">Percent values.</Property>
  <Property name="duration" type="string">Time spans.</Property>
  <Property name="custom" type="any">User-defined schema.</Property>
</Properties>

# Testing & best practices

## Test your Flows

<Tabs>
  <Tab title="Simulation">
    <Steps>
      <Step title="Upload a recording">
        Validate recognition and branch logic with real audio.
      </Step>
      
      <Step title="Simulate variables">
        Seed test values for <code>
        @name</code>

        , <code>
        @phone</code>

        , etc.
      </Step>
      
      <Step title="Path testing">
        Walk each branch and confirm expected prompts.
      </Step>
      
      <Step title="Error testing">
        Force empty/malformed inputs to validate fallbacks.
      </Step>
    </Steps>
  </Tab>
  <Tab title="Live">
    <Steps>
      <Step title="Use test numbers">
        Route calls to non-production DIDs.
      </Step>
      
      <Step title="Monitor in real time">
        Watch transcripts and events in the dashboard.
      </Step>
      
      <Step title="Measure performance">
        Track response times, success rates, transfers.
      </Step>
      
      <Step title="A/B versions">
        Compare alternative prompts/flows.
      </Step>
    </Steps>
  </Tab>
</Tabs>

## Design guidelines

<Accordion>
  <AccordionItem title="Conversation design">
  Keep prompts natural and concise, offer clear next steps, plan for interruptions, and confirm critical data before proceeding.
</AccordionItem>

  <AccordionItem title="Technical hygiene">
  Test all branches, add fallback edges, watch performance metrics, and keep external dependencies resilient.
</AccordionItem>

  <AccordionItem title="Variable management">
  Use descriptive names, validate inputs, handle missing data gracefully, and scrub sensitive data when not needed.
</AccordionItem>
</Accordion>

# Troubleshooting guide

<Accordion>
  <AccordionItem title="Flow not responding">
  Verify block connections, confirm required variables are set, test blocks individually, and check API auth/endpoints.
</AccordionItem>

  <AccordionItem title="Poor call quality">
  Review Talk block voice settings, try alternate voices, adjust rate and pauses, and check variable interpolation.
</AccordionItem>

  <AccordionItem title="Transfer failures">
  Confirm destination numbers/agents, test fallbacks, and verify timing and availability windows.
</AccordionItem>

  <AccordionItem title="Variable issues">
  Check spelling/case, ensure values are set before use, validate data types, and test prompts using variables.
</AccordionItem>

  <AccordionItem title="API integration problems">
  Test endpoints in isolation, verify credentials and headers, confirm request/response schemas, and watch rate limits/timeouts.
</AccordionItem>
</Accordion>
